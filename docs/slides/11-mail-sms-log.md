# 목적 & 핵심 가치

### 왜 만드나?

모든 이메일·SMS·알림톡 발송 이력을 **MailLog 엔티티**로 일원화하여 추적하고, 발송 실패 건의 재시도와 재발송을 시스템에서 직접 처리한다. 도메인 규칙상 **발송 시 반드시 MailLog 기록**이 생성되어야 한다.

| 구분 | 기존 시스템 | 신규 시스템 |
|------|------------|------------|
| 발송 이력 | 이메일·SMS 별도 로그, 알림톡 이력 미보관 | MailLog 단일 테이블로 통합 관리 |
| 발송 실패 | 실패 확인 어려움, 수동 재발송 | 실패 건 목록 + 원클릭 재시도 |
| 검색·필터 | 채널별 별도 관리 화면 | 통합 필터 (기간/채널/수신자/상태) |
| 감사 추적 | 발송 기록 누락 가능 | 모든 발송에 MailLog 자동 생성 보장 |

# 시나리오 1: 자동 발송 + MailLog 기록

```
시스템 (A/S 상태 변경)       MailLog               SMTP/SMS 게이트웨이
        │                     │                        │
        ├─ 상태 변경 트리거 ───→│                        │
        │  (A/S 접수 완료)     │                        │
        │                     ├─ MailLog 레코드 생성     │
        │                     │  (상태: 대기)            │
        │                     │                        │
        │                     ├─ 발송 요청 ────────────→│
        │                     │  (이메일 + SMS 동시)     │
        │                     │                        │
        │                     │←── 이메일 발송 성공 ─────┤
        │                     ├─ MailLog 상태→성공       │
        │                     │                        │
        │                     │←── SMS 발송 성공 ────────┤
        │                     ├─ MailLog 상태→성공       │
        │                     │                        │
        │  ※ 도메인 규칙:       │                        │
        │  발송 없이 MailLog    │                        │
        │  누락은 허용 안 됨     │                        │
```

# API 엔드포인트 매핑

| 메서드 | 엔드포인트 | 설명 | 권한 |
|--------|-----------|------|------|
| GET | `/api/mail-logs` | 발송 이력 목록 조회 (필터·페이징) | 전체 (본인 건 / 팀장 이상 전체) |
| GET | `/api/mail-logs/{id}` | 발송 이력 상세 조회 | 전체 |
| POST | `/api/mail-logs/resend/{id}` | 동일 내용 재발송 + 새 MailLog 생성 | A/S 담당자 이상 |
| POST | `/api/mail-logs/resend/{id}/modify` | 수신자 수정 후 재발송 | A/S 담당자 이상 |
| GET | `/api/mail-logs/stats` | 발송 통계 (채널별/상태별/기간별) | 팀장 이상 |
| GET | `/api/mail-logs/failed` | 실패 건 목록 조회 | 전체 (본인 건) |

# 확인 필요 사항

1. 알림톡 발송 이력 기존 데이터 마이그레이션 범위
2. MailLog 보관 기간 정책 (영구? 12개월?)
3. 자동 재시도 횟수 제한 (실패 시 최대 N회 자동 재시도?)
4. 대량 발송 건의 MailLog 처리 방식 (개별 건 vs 배치 건)
5. 발송 내용 개인정보 마스킹 정책 (조회 시 부분 마스킹 필요?)
