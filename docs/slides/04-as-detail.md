# 04. A/S 접수 및 상세 관리

**접수 → 배정 → 진행 → 완료 전 과정 통합 관리**

Phase 1 산출물 | 슬라이드 04 / 15

---

## 목적 & 핵심 가치

### 왜 만드나?

- A/S 접수부터 종료까지 **전체 라이프사이클을 한 화면에서 관리**
- 고객 전화번호 입력만으로 **고객·제품 정보 자동 매칭**
- 상태 변경 이력(타임라인)으로 **처리 과정 투명하게 추적**

### 기존 vs 신규

| 기존 어드민 | 신규 통합 앱 |
|------------|-------------|
| 접수 폼과 상세가 분리 | 접수 + 상세 + 이력 통합 |
| 상태 변경 이력 없음 | 타임라인으로 전 이력 자동 기록 |
| 비용 별도 관리 | 상세 내 비용 3축 입력 (지출/수납/보상) |
| 사진 첨부 불가 | 증상 사진 최대 5장 첨부 |
| 담당자 변경 이력 없음 | 담당자 변경 시 이력 자동 기록 |

## 접수 폼 [A] 상세

| UI 요소 | 타입 | 상세 |
|---------|------|------|
| 전화번호 | text | **필수**, 입력 후 [자동매칭] → 고객 정보 자동 채움 |
| 제품 선택 | select | **필수**, 해당 고객 구매 제품 목록 (모델명·시리얼 표시) |
| 증상 분류 | select | **필수**, 하드웨어/소프트웨어/외관/기타 → 하위 분류 연동 |
| 증상 상세 | textarea | **필수**, 최대 1000자 |
| 사진 첨부 | file | 선택, 최대 5장, jpg/png, 장당 10MB |
| 접수 채널 | select | 전화/이메일/방문/온라인 |
| 우선순위 | select | 낮음/보통/높음/긴급 (기본: 보통) |
| 희망 완료일 | datepicker | 선택 |
| 접수 저장 | btn-primary | 유효성 검증 후 저장 → 상세 페이지로 이동 |

## 상태 흐름

```
접수 ──→ 배정 ──→ 진행 ──→ 완료 ──→ 종료
          │        │
          │        └──→ 보류 ──→ 진행 (재개)
          │
          └──→ (팀장/총괄이 담당자 지정)
```

### 상태 정의

| 상태 | 설명 | 변경 가능 역할 |
|------|------|---------------|
| **접수** | A/S 접수 완료, 담당자 미배정 | A/S 담당자, 총괄 |
| **배정** | 담당자 지정 완료 | 팀장, 총괄 |
| **진행** | 수리/처리 진행 중 | A/S 담당자, 팀장, 총괄 |
| **보류** | 부품 대기, 고객 연락 불가 등 | A/S 담당자, 팀장, 총괄 |
| **완료** | 처리 완료, 고객 확인 대기 | A/S 담당자, 팀장, 총괄 |
| **종료** | 고객 확인 완료, 건 마감 | 팀장, 총괄 |

## 타임라인 [C] & 비용 입력 [E]

### 타임라인 자동 기록 항목

| 이벤트 | 기록 내용 |
|--------|----------|
| 상태 변경 | 변경 전→후, 변경자, 일시 |
| 담당자 변경 | 이전→신규 담당자, 변경자, 일시 |
| 처리 메모 추가 | 메모 내용, 작성자, 일시 |
| 비용 입력/수정 | 비용 항목, 금액, 변경자, 일시 |
| 이메일/SMS 발송 | 발송 대상, 발송자, 일시, MailLog ID |

### 비용 3축 입력

| 축 | 설명 | 예시 |
|----|------|------|
| **지출** (우리 지출) | 부품비, 인건비, 택배비 등 | 부품비 35,000원 |
| **수납** (고객 수납) | 고객에게 청구한 수리비 | 수리비 50,000원 |
| **보상** (본사 보상) | 본사로부터 보상받는 금액 | 보증 보상 35,000원 |

## 시나리오 SC-1: A/S 담당자 신규 접수

```
A/S 담당자 로그인
    ↓
A/S 접수 페이지 진입 (또는 고객조회(#1)에서 [A/S 접수] 클릭)
    ↓
전화번호 입력 → [자동매칭]
    → API: GET /api/customers?phone=010-1234-5678
    → 고객명, 전화번호 자동 채움
    ↓
제품 선택 → 모델명·시리얼 자동 표시
    ↓
증상 분류: 하드웨어 > 전원부
증상 상세: "전원 버튼 눌러도 켜지지 않음"
사진 첨부: 2장
    ↓
[접수 저장] 클릭
    → API: POST /api/tickets
    → 상태: "접수", 타임라인 첫 항목 생성
    → 상세 페이지로 이동
```

## 시나리오 SC-2: 팀장 배정 + 담당자 처리 + 완료

```
팀장 로그인 → A/S 현황(#5)에서 미배정 건 확인
    ↓
AS-20260428-001 클릭 → 상세 페이지
    ↓
[담당자 변경] → 이상현 선택
    → API: PATCH /api/tickets/{id} { "assignee": "이상현" }
    → 상태: 접수 → 배정, 타임라인 기록
    ↓
--- A/S 담당자(이상현) ---
    ↓
상세 페이지 진입 → [진행] 상태 변경
    → API: POST /api/tickets/{id}/history { "status": "IN_PROGRESS" }
    ↓
처리 메모: "전원부 PCB 교체 완료"
    → API: POST /api/tickets/{id}/history { "type": "MEMO", "content": "..." }
    ↓
비용 입력: 지출 35,000 / 수납 50,000 / 보상 35,000
    → API: PATCH /api/tickets/{id} { "costs": {...} }
    ↓
[완료] 클릭 → 고객에게 [SMS 발송]
    → POST /api/mail-logs/sms → MailLog 자동 기록
    → 토스트: "SMS 발송 완료"
```

## API 엔드포인트 매핑

| 액션 | Method | Endpoint |
|------|--------|----------|
| A/S 접수 | POST | `/api/tickets` |
| 티켓 상세 조회 | GET | `/api/tickets/{id}` |
| 티켓 수정 (상태·담당자·비용) | PATCH | `/api/tickets/{id}` |
| 상태 이력 추가 | POST | `/api/tickets/{id}/history` |
| 타임라인 조회 | GET | `/api/tickets/{id}/history?page&size` |
| 사진 업로드 | POST | `/api/tickets/{id}/attachments` |
| 사진 조회 | GET | `/api/tickets/{id}/attachments` |
| 이메일 발송 | POST | `/api/mail-logs/email` |
| SMS 발송 | POST | `/api/mail-logs/sms` |

# 확인 필요 사항

- **증상 분류 체계**: 하드웨어/소프트웨어/외관/기타 하위 분류 목록 → 실무진 작성 필요
- **보류 사유 분류**: 부품 대기/고객 연락 불가/기타 → 정형화 여부 팀장 확정
- **종료 조건**: 고객 확인 방법 (전화 확인? 자동 7일 후 종료?) → 총괄 결정
- **비용 입력 필수 여부**: 완료 전 비용 입력 필수 or 선택 → 팀장/총괄 확정
